The question

Async work (overnight scrapers, long-running jobs) breaks hourly billing. Token-cost reflects active intensity but ignores invisible work. What if the agent billed JB directly, JB paid out-of-pocket, and reimbursed himself from the client? Does that fix the accountability gap — or just create a worse one?

Also: 3 of 4 Titan clients (Connective Sports, Tatlist, Marian Washington) are or should be on SLAs. Per-session billing is the minority case — the primary billing instrument should be a monthly retainer with one of the four per-session models for overages.

The four models, side-by-side

1. Hourly (wall time)

Industry default

duration_ms × $rate

  • Universal frame — every client knows hourly.
  • Trivially auditable, calendar-shaped.
  • Predictable for the client.
  • Async kills it. Overnight scraper = 8 indefensible hours.
  • Anti-parallelism — four tools at once bills like one.
  • Idle time (CI, deploys) bills you, not the client.
  • No signal for steering intensity.

2. Token × N

What the statusline currently shows

cost_usd × multiplier

  • Async-immune — only counts when the model is working.
  • Captures intensity, steering, model choice.
  • Parallel-friendly. Self-scaling.
  • One-line auditable from the snapshot.
  • Multiplier feels arbitrary without an anchor.
  • Misses invisible work — calls, planning, post-fixes.
  • Bills move with Anthropic's prices.
  • "What's a multiplier?" → client confusion.

3. Tiered hourly (active time only)

Hourly that doesn't punish you for sleeping

api_duration × tier_rate per session

  • Looks like normal consulting hourly to the client.
  • Async-immune — only the active portion counts.
  • Lets you charge more for strategic work (refactors, architecture) and less for boilerplate.
  • Easy mental model for both sides.
  • Tier assignment is judgment — open to dispute.
  • Need to label every session (the drag bit). Friction.
  • API duration isn't quite "effort" — fast tools-only sessions undercount steering.
  • Still doesn't catch off-Claude work without manual line items.

4. SLA / Retainer

The shape 3 of 4 clients are moving toward

Monthly recurring fee, response-time gates, scope cap; overages billed at one of the above primitives.

  • Predictable MRR. Revenue doesn't stop when the build does.
  • Response-time guarantees are a real product, not a side effect — Aisha can sell it.
  • Client commits ahead of time → less per-session negotiation friction.
  • Stripe subscription handles the recurring side natively; overages stay as one-off invoices.
  • Scope definition is real work — what's in the retainer vs. what's overage?
  • Client may feel they "overpaid" on quiet months. Mitigate with carryover hours or banked credits.
  • Contracts get heavier — SLA terms, escalation paths, breach remedies.
  • Doesn't replace per-session models for overages — you still need one of the four.

5. Agent → JB → client

JB's wildcard idea

Agent bills JB at token × N. JB bills client at agent_bill × markup.

  • Skin-in-the-game. JB won't run sessions he wouldn't pay for himself. Forces scope discipline.
  • Quality gate. Bad sessions get filtered before they hit the client.
  • Real-money pause. Wallet empty? No work runs. "Fair business" enforced mechanically.
  • Three-way alignment — agent serves JB (paying customer), JB serves client (real reimbursement risk).
  • Markup justifies itself — covers judgment, risk absorption, float cost.
  • Capital tied up until reimbursement. Cashflow burden on JB.
  • Client could refuse reimbursement — JB eats the loss.
  • Two invoices to track. Accounting complexity.
  • Slower client turnaround (extra hop).
  • "I have to pay before my agent works for me" can feel adversarial if not framed right.
  • Tax pass-through implications — talk to an accountant.

Deep dive — agent bills JB, JB reimbursed by client

The accountability claim

"No work gets done until fair business is being done." Three places this enforces itself:

  • Before starting. Pre-flight estimate; if wallet is empty or scope is fuzzy, agent declines until it's resolved.
  • During work. Live running tab in the statusline. JB sees the meter and can stop the agent before runaway.
  • After delivery. Until the client's prior invoice clears, the wallet refills slowly. New work for that client slows down — natural pressure.

Implementation suggestions, ranked by how far down the rabbit hole

a. Light — pre-flight estimate gate

At session start, agent estimates token cost based on past sessions in the same repo (median × multiplier). Shows: "This will probably cost ~$X.XX. You'll bill the client ~$Y.YY. Continue?"

Lift: small. Just a wrapper hook on session start. Catches the worst overruns without infrastructure.

b. Medium — wallet ledger

A real balance in ~/.claude/billing/wallet.json. Topped up when client reimbursement clears (manual update or Stripe webhook). Debited per session. Statusline shows balance. Empty wallet = polite refusal.

Lift: medium. Needs wallet schema, top-up flow, debit hook, and a Stripe payment_received webhook to auto-refill. But this is the model that mechanically enforces "no work without fair business."

c. Medium — tiered authorization

Sessions under $5 estimated cost: auto-run. $5–30: confirm via AskUserQuestion. $30+: require an SOW reference (a string the client agreed to in advance, e.g. ticket ID).

Lift: medium. Useful even without the wallet — keeps you honest about scope.

d. Heavy — two-sided ledger

Track both agent→JB invoices AND JB→client invoices in one view. Reconciliation report shows per-client P&L: cost from agent, billed to client, paid by client, outstanding float. This is what shows up in the HTML artifact JB shares with the client.

Lift: heavy. But this is the dashboard that closes the loop — turns "I think we're profitable on Aisha's account" into a number.

e. Heavy — reimbursement gating

Agent refuses new work for client X if JB's invoice to X is > N days past due. Combined with the wallet, this means the system stops digging the hole deeper when a client goes slow on payments.

Lift: heavy. Needs the two-sided ledger first. But this is what turns "no work without fair business" from a slogan into a property.

Open questions for Aisha

  1. SLA structure across the book. Connective Sports and Tatlist are about to be on SLAs; Marian Washington should be but isn't. What's in the retainer (response time, included hours, scope) vs. out (overage rate, ad-hoc requests)? And what's the catch-up plan for Marian — is it a separate conversation or part of a "we're formalizing SLAs across the board" rollout?
  2. Is the markup defensible to the client? 4× sounds spicy in the abstract, but it's covering: judgment, risk absorption (you might never get reimbursed for a bad session), and the float (your money tied up). What multiplier does the math say is fair?
  3. Disclosure. Do clients know the agent costs $X and they're paying Y? Or do they just see "engineering hours"? Either is defensible — pick deliberately.
  4. Wallet seed. If we go wallet-based, what's the opening balance? Self-funded float, retainer from each client, or pure pay-as-you-go that refuses every first session?
  5. Failed reimbursement policy. When a client refuses (or just doesn't pay), what happens? Write off, send to collections, blacklist, escalate? This is the bit that turns "skin in the game" from a feature into a liability.
  6. Tax treatment. Pass-through structure changes whether the agent's cost is a deductible expense for JB. Worth one call with the accountant before any of this is real.

Recommendation if we have to ship one thing tomorrow

For the three SLA-bound clients (Connective Sports, Tatlist, Marian Washington): monthly retainer via Stripe subscription covers the baseline; overages get a one-off invoice using Token × N (agent → JB) + tiered hourly (JB → client), drag-tier-assigned per session in this UI.

For GapSnow (ad-hoc nonprofit): per-session tiered hourly with the 0.65× modifier. No retainer.

Implementation order: (a) pre-flight estimate first. (b) Stripe subscription wiring for the SLA shift. (c) wallet ledger if/when a client where it matters needs it. (d)/(e) only if the first three are working and we still feel exposed.
